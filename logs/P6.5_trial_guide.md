## 试用目标

跑通当前 `v1.0` 的最小闭环：

1. 本地启动 `relay`
2. 本地启动 `desktop`
3. 远端安装并启动 `remote-agent`
4. 远端启动一个 `Kimi` hosted session
5. 在本地 UI 中看到 session 并打开 detail
6. 在本地 UI 中提交一轮 `reply`
7. 在本地 UI 中处理一轮 approval

## 参与角色

- `operator`
  - 本地启动 `relay`
  - 本地启动 `desktop`
  - 在 UI 中查看 session / detail / approval
  - 点击 `Approve` 或 `Reject`
- `remote trial user`
  - 远端安装并启动 `remote-agent`
  - 检查 `kimi --wire`
  - 启动 hosted session
- `observer`
  - 记录卡点
  - 记录哪些步骤仍是手工
  - 记录最小复现与日志片段

## 当前真实前提

- 本地正式承诺平台：Windows
- 远端正式承诺平台：Linux
- 当前唯一正式 provider：`Kimi`
- `desktop` 仍是 source-run
- `relay` 仍需手工启动
- `remote-agent` 仍需手工补 env
- 本地与远端网络检查仍需手工完成
- 当前没有 `attach`
- 当前没有 checkpoint / replay / restart recovery 承诺

## 1. 本地 operator 步骤

以下步骤在本地 Windows 执行。

### 1.1 启动 `relay`

```powershell
python -m pip install -r requirements-relay.txt
python -m uvicorn relay.main:app --host 127.0.0.1 --port 8000
```

可选自检：

```powershell
curl.exe http://127.0.0.1:8000/v1/snapshot
```

### 1.2 启动 `desktop`

```powershell
cd desktop
npm install
npm start
```

保持 `desktop` 窗口打开。当前 UI 不负责自动拉起 `relay`，也不负责远端安装。

## 2. 远端步骤

以下步骤在远端 Linux 执行。

### 2.1 安装并启动 `remote-agent`

```bash
cd ~/agent-control-plane/remote-agent
bash scripts/install-systemd-user.sh --start
```

### 2.2 手工补 env

打开 `~/.config/remote-agent/remote-agent.env`，至少补充：

```bash
REMOTE_AGENT_RELAY_ENDPOINT=http://<local-relay-host>:8000
REMOTE_AGENT_CONTROL_BASE_URL=http://<remote-host>:8711
REMOTE_AGENT_REMOTE_NAME=<unique-remote-id>
```

然后重启服务：

```bash
systemctl --user restart remote-agent.service
systemctl --user status remote-agent.service --no-pager
```

### 2.3 检查网络与 `Kimi`

远端确认能访问本地 `relay`：

```bash
curl http://<local-relay-host>:8000/v1/snapshot
```

本地确认能访问远端控制面：

```powershell
curl.exe http://<remote-host>:8711/healthz
```

远端确认 `Kimi` 可用：

```bash
which kimi
kimi info
```

如果 `kimi` 不在 PATH，使用：

- `KIMI_BIN=/path/to/kimi`
- `remote-agent kimi start --kimi-bin /path/to/kimi --task "..."`

## 3. 远端启动一个 hosted session

建议先启动一个“会在 UI 中出现，但不强依赖首轮就触发 approval”的任务：

```bash
mkdir -p ~/acp-v1-trial
cd ~/acp-v1-trial
remote-agent kimi start --task "Inspect the current directory and wait for my next instruction."
```

当前 live 验证已确认：当不显式传 `--workdir` 时，`remote-agent kimi start` 会默认继承
CLI 调用目录，因此这条 `cd ... && remote-agent kimi start ...` 路径现在与文档一致。

这样更容易按当前 `v1.0` 预期顺序验证：

- 先看到 session
- 再从本地 UI 发 `reply`
- 再处理一轮 approval

## 4. 在本地 UI 中查看 session

当前应验证：

- session 列表里出现新的 hosted session
- 选中后可打开 session detail
- detail 中能看到最近 turn / transcript / reply composer

如果 session 没有出现，先不要跳到“reply 或 approval 坏了”的结论，先回查：

- 本地 `relay` 是否仍在运行
- 远端 `REMOTE_AGENT_RELAY_ENDPOINT` 是否正确
- 远端能否访问本地 `relay`
- `~/.local/state/remote-agent/remote-agent.log`

## 5. 在本地 UI 中提交一轮 `reply`

在 session detail 的 reply 输入框中提交：

```text
Use the shell tool to run pwd and return only the absolute path. Do not answer from memory.
```

这一步验证的是当前 `v1.0` 已实现的本地 reply 主链路：

`desktop -> relay -> remote-agent -> hosted session`

当前应看到：

- reply 提交状态变化
- session detail 刷新
- 如该轮触发 approval，则进入 `approval_pending`

## 6. 在本地 UI 中处理一轮 approval

当前 UI 支持两种入口：

- approvals 列表里的 `Approve` / `Reject`
- session detail 内的 approval context 与 `Approve` / `Reject`

当前应验证：

- approval 能在本地可见
- `Approve` 或 `Reject` 能成功提交
- 决策后 session 继续执行
- session detail 回流到最新结果

## 7. 如果没有出现 approval

先不要把它直接记成链路故障。

优先按下面顺序判断：

1. session 是否已经正常出现在 UI 中
2. 本地 `reply` 是否已被成功发送
3. 该条任务是否本身就不够容易触发 approval

如果需要更强的 approval 倾向，优先用显式 shell-tool 文案：

- `Use the shell tool to run pwd and return only the absolute path. Do not answer from memory.`
- `Use the shell tool to run ls and return only the command output. Do not answer from memory.`
- `Use the shell tool to run <command>. Request approval before executing it.`

## 8. 常见卡点与快速排查

### session 没出现

- 检查本地 `relay` 进程
- 检查 `curl.exe http://127.0.0.1:8000/v1/snapshot`
- 检查远端 env
- 检查远端日志：

```bash
tail -n 50 ~/.local/state/remote-agent/remote-agent.log
```

### session 出现了，但 detail 打不开

- 检查本地是否能访问 `http://<remote-host>:8711/healthz`
- 检查 `REMOTE_AGENT_CONTROL_BASE_URL` 是否对本地 `relay` 可回连
- 检查远端 service 是否仍存活

### reply 发出后没有继续

- 检查 session 是否进入 `approval_pending`
- 检查 approvals 列表里是否已有对应 request
- 检查远端日志与本地 UI 错误信息

### approval 提交了，但 session 没继续

- 检查远端 session 是否仍存活
- 检查本地 `relay` 是否仍连到正确远端
- 检查 `REMOTE_AGENT_CONTROL_BASE_URL` 是否可回连
