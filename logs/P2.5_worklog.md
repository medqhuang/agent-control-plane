This document records the closure work completed in `P2.5 Kimi Bridge Hardening`.

- freeze a stable bridge-level `session_id / request_id` derivation rule
- freeze a stable minimal `provider_writeback` shape
- keep Kimi-specific details inside `adapters/kimi/`

- verify the hardening on the real `WSL -> SSH -> zhaojin.ustc.edu.cn` path

## Completed Work

### 1. Remote Bridge Identifier Rules Frozen

The remote-backed Kimi bridge now uses stable derivation rules.

Remote `session_id` rule:

- `kimi_remote_<normalized_suffix>`

Normalization rule:

- lowercase
- replace any non-alphanumeric run with `_`
- strip leading and trailing `_`

Example:

```text
approve_real_20260331 -> kimi_remote_approve_real_20260331
```

Remote `request_id` rule:

- `kimi_request__<normalized_remote_host>__<session_id>__seq_<seq>`

Example:

```text
kimi_request__zhaojin_ustc_edu_cn__kimi_remote_approve_real_20260331__seq_1
```

These rules keep the minimal bridge correlation stable across:

- remote host
- session id
- request id
- source sequence

### 2. Minimal Provider Writeback Contract Frozen

The minimal `provider_writeback` structure is now fixed to carry:

- `type`
- `provider`
- `remote`
- `session_id`
- `request_id`
- `source_seq`
- `decision`
- `provider_action`
- `mode`
- `result`

Optional evidence field:

- `capture_excerpt`

Minimal real-remote example:

```json
{
  "type": "approval_response_writeback",
  "provider": "kimi",
  "remote": "zhaojin.ustc.edu.cn",
  "session_id": "kimi_remote_approve_real_20260331",
  "request_id": "kimi_request__zhaojin_ustc_edu_cn__kimi_remote_approve_real_20260331__seq_1",
  "source_seq": 1,
  "decision": "approve",
  "provider_action": "approve_request",
  "mode": "real_tmux_smoke",
  "result": "real_tmux_ok"
}
```

This keeps the bridge contract stable without changing the existing relay
snapshot or approval-response response model.

### 3. Commit Order Hardened For Remote-Backed Approvals

Before this hardening, local relay state could be updated before the provider
writeback result was known.

The current order for remote-backed Kimi approvals is now:

1. read current `approval` and `session`
2. reserve the next relay event sequence
3. execute provider writeback
4. only after writeback succeeds:
   - update `approval.status`
   - update `session.status`
   - append one `approval_response` event

If provider writeback fails or times out:

- local `approval` remains `pending`
- local `session` remains `waiting_approval`
- no new `approval_response` event is appended
- the endpoint returns a clear error response

### 4. Failure Contract Added

Two failure classes are now explicit:

- `HTTP 502`
  - provider writeback failed
- `HTTP 504`
  - provider writeback timed out

Representative error response:

```json
{
  "detail": {
    "message": "provider writeback timed out",
    "provider": "kimi",
    "remote": "zhaojin.ustc.edu.cn",
    "session_id": "kimi_remote_timeout_real_20260331",
    "request_id": "kimi_request__zhaojin_ustc_edu_cn__kimi_remote_timeout_real_20260331__seq_1",
    "decision": "approve",
    "source_seq": 1,
    "reason": "forced remote Kimi writeback failure: ..."
  }
}
```

For validation only, a helper-only environment switch was added:

- `KIMI_BRIDGE_FORCE_WRITEBACK_FAILURE=timeout`
- `KIMI_BRIDGE_FORCE_WRITEBACK_FAILURE=error`

This switch remains adapter-local and is not exposed as a new API surface.

## Remaining Bridge Risks

The current bridge is stronger than P2, but it is not production-ready.

Known limitations still present:

- bridge `request_id` is still adapter-derived, not a Kimi-native request id
- writeback still depends on `tmux` key injection and current Kimi TUI layout
- reject confirmation still lacks a Kimi-native structured receipt
- local relay state is still in-memory only
- if remote writeback succeeds and the relay process crashes immediately after,
  provider and relay state can still diverge
- no multi-remote coordination yet
- no durable replay, reconciliation, or recovery logic yet
- Windows-side `capture_excerpt` can still contain encoding noise

These are known bridge risks, not hidden regressions.
