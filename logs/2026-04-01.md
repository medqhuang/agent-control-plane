- 审读 `README.md`、`DEV.md`、`relay/` 与 `adapters/kimi/adapter.py`，确认当前项目实现原理。
- 核心结论：
  - `snapshot` 路径是轻量 in-memory 读取，正常不会构成明显卡顿。
  - 当前主要性能风险在 `approval-response` 的远端 writeback：本地 FastAPI -> Kimi adapter -> WSL -> bash -> SSH -> tmux -> 轮询远端 TUI。
  - 小规模单用户场景可接受，但并发审批增多时会出现明显等待，不应把当前 bridge 形态表述为低延迟或生产级集成。
- 后续建议：
  - `P3` 先做本地控制端，不要顺手重构 bridge。
  - 等 `P3` 稳定后，再考虑把 provider writeback 从同步阻塞链路改成后台任务或更结构化协议。
- 追加审读结论：
  - 当前项目的实际使用方式是“先启动 relay，再用 demo_push 或 remote_smoke 向 relay 注入 Kimi approval 事件，最后通过 snapshot 或 desktop 读取状态”。
  - 现阶段 desktop 仍是只读 snapshot viewer，还没有真正的 approve / reject 按钮；审批提交仍需直接调用 `POST /v1/approval-response`。
