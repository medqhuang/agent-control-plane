# P4.5 Worklog

Updated: 2026-04-01
Status: Complete

This document records the closure work completed in `P4.5 Hosted Session Usability`.
The scope is limited to making the single-remote hosted-session path actually usable,
closing the relay loop, adding the minimal session CLI, freezing the hosted-session
contract, and freezing the recovery contract without claiming that the recovery system
is already implemented.

## Stage Goal

P4.5 was created to close the gap between:

- “the remote-agent foundation exists”

and

- “a single remote hosted session can be used and controlled day to day”

The stage goal was therefore:

- complete the `remote-agent -> relay` event path
- complete the `relay -> remote-agent -> provider` approval decision path
- add the minimal hosted-session CLI
- freeze the usage contract for hosted sessions
- freeze the recovery contract without over-claiming recovery implementation

## Completed Work

### 1. P4.5-A Relay Integration Completed

The remote hosted-session path now reports provider-agnostic standard events back to relay.

Completed pieces:

- `remote-agent -> relay` standard event reporting
- relay-side snapshot visibility for remote sessions
- relay-side pending approval visibility for remote approvals
- ordered session events with sequence information
- approval correlation through stable `request_id`
- `relay -> remote-agent -> kimi --wire` approval writeback

The key correctness rule remained intact:

- first provider writeback succeeds
- then relay commits local approval and session state

Failure writeback does not pollute snapshot state.

### 2. P4.5-B Session CLI Completed

The remote shell now has the minimum hosted-session management commands:

- `remote-agent sessions`
- `remote-agent watch <session_id>`
- `remote-agent reply <session_id> --message "..."`
- `remote-agent stop <session_id>`

These commands work against hosted sessions inside `remote-agent serve`, not against
one-shot worker invocations.

This made the remote path usable without falling back to ad-hoc provider-side shells.

### 3. P4.5-C Hosted Session Contract Completed

The hosted-session usage contract is now fixed in project documentation.

The stage closed the ambiguity around:

- `start` as hosted background creation
- `sessions` as current runtime list only
- `watch` as single-read, not streaming follow
- `reply` as non-attach turn continuation
- `stop` as idle-only stop
- `attach` as explicitly unsupported today

This matters because the CLI now exists and must not be documented as more powerful
than the current implementation.

### 4. P4.5-D Recovery Contract Completed

The minimal recovery contract is now written down across the project docs.

The contract now clearly distinguishes:

- service revival
- control-plane state restoration
- provider execution-state restoration

It also fixes the current boundaries for:

- local desktop disconnect
- `remote-agent` restart
- relay restart
- provider subprocess exit
- checkpoint schema expectations
- pending approvals replay semantics
- control-plane event replay semantics

The important closure here is wording discipline:

`P4.5-D` completed the contract, not the recovery implementation.

## Session CLI Current Boundaries

The current Session CLI remains intentionally minimal:

- `remote-agent sessions` only shows sessions still hosted in the current `remote-agent` process
- `remote-agent watch <session_id>` is a single latest-state read, not a follow mode
- `remote-agent reply <session_id> --message "..."` appends one more turn without attaching a TTY
- `remote-agent stop <session_id>` works only when the session is idle
- `stop` rejects:
  - `approval_pending`
  - a turn still in progress
- `remote-agent attach <session_id>` is not implemented

These are current behavior boundaries, not future-looking suggestions.

## Recovery Contract Current Boundaries

The current recovery contract is intentionally narrower than a full recovery system.

Current implemented and documented boundaries:

- remote hosted sessions continue after local desktop disconnect as long as remote `remote-agent serve` and the provider subprocess are still alive
- `systemctl --user` can revive the `remote-agent` service itself
- relay and remote-agent both still keep their live control-plane state in memory
- `request_id` remains the stable approval key
- session events keep ordering information

Current contract-only, not-yet-implemented recovery items:

- checkpoint persistence
- checkpoint restore
- pending approvals replay after restart
- control-plane event replay after restart
- `remote-agent` restart restoration of prior hosted sessions
- provider execution-scene `resume / reattach`

This distinction is the main reason `P4.5` can be marked complete without falsely
claiming that the recovery system is done.

## Implemented Capability vs Contract-Only Boundary

Implemented by the end of P4.5:

- long-running remote `remote-agent serve`
- direct remote `kimi --wire` start path
- relay snapshot visibility for remote sessions and approvals
- relay approval decision writeback to hosted Kimi sessions
- hosted-session CLI:
  - `sessions`
  - `watch`
  - `reply`
  - `stop`
- hosted-session usage contract
- recovery contract wording

Contract-only by the end of P4.5:

- recovery-state labels such as `online / offline / awaiting_reconnect`
- minimal checkpoint schema as a future persisted artifact
- pending approvals replay
- control-plane event replay
- `remote-agent` restart restore
- provider execution-state restore

## Why P5 Can Start

P5 can start because the single-remote hosted-session path is now stable enough to
act as the baseline for multi-remote work.

The main prerequisites are already closed:

- remote execution boundary is at `remote-agent`
- remote service hosting is real, including logout survival on the target host
- provider startup uses `kimi --wire`
- relay can see remote session and approval state
- approval decisions can round-trip back to the hosted Kimi session
- the remote shell can list, inspect, continue, and stop hosted sessions
- the hosted-session and recovery wording is no longer ambiguous

That is sufficient to move from “single remote usable path” to “multi-remote aggregation”.

## Why P8 Still Keeps Recovery Implementation

P8 still owns the actual recovery system because the hard reliability work remains open:

- persisted checkpoint storage
- persisted approval queue recovery
- persisted event replay
- remote-agent restart restoration
- relay restart state rehydration
- reconnect hardening
- provider-dependent execution-scene recovery

P4.5 closed the meaning of those features.
P8 remains the stage that must implement them.

## Current Boundaries After P4.5

The stage is complete, but the following boundaries remain explicit:

- multi-remote aggregation is not implemented yet
- `attach` is not implemented yet
- relay remains in-memory
- remote-agent runtime remains in-memory
- recovery remains contract-defined rather than fully implemented
- provider execution-scene recovery remains provider-dependent future work

P4.5 is therefore closed as a usability stage, not as a reliability-completion stage.
