## Release Unit

The first public Beta is defined as a technical source-run / source-install
Beta.

The release unit is the current repo, with these three user-visible runtime
surfaces:

- local `relay/`
- local `desktop/`
- remote `remote-agent/`

This is intentionally narrower than a polished product release. The first
public Beta does not promise an installer, a packaged daemon, or a hosted
service.

## Included Components

- local `relay`
- local `desktop`
- remote `remote-agent`
- `Kimi --wire`
- hosted session CLI:
  - `remote-agent kimi start --task "..."`
  - `remote-agent sessions`
  - `remote-agent watch <session_id>`
  - `remote-agent reply <session_id> --message "..."`
  - `remote-agent stop <session_id>`
- multi-remote baseline in the local desktop control plane

## Explicitly Excluded Components

- `Codex`
- `Claude Code`
- `attach`
- reconnect
- checkpoint persistence
- pending approvals replay
- control-plane event replay
- `remote-agent` restart restoration of hosted in-memory state
- provider execution-state recovery
- cloud relay / hosted relay
- desktop installer
- `remote-agent` PyPI package, wheel release, or OS package release

## Delivery Shape

### desktop

- delivered as repo source under `desktop/`
- current launch shape is `npm install && npm start`
- not an installer
- not a signed desktop distribution

## Desktop Delivery Baseline

- desktop remains a repo-local source-run surface for the first public Beta
- the launch path is:
  - `cd desktop`
  - `npm install`
  - `npm start`
- `desktop/package.json` currently exposes only `start` and `dev`
- there is no formal build, package, or installer script in the current
  desktop surface
- the first public Beta desktop path remains local Windows only
- desktop requires a separately started local `relay`
- desktop defaults to `http://127.0.0.1:8000` and can be redirected only
  through `RELAY_BASE_URL`

## Why Source-Run Is Acceptable

- the first public Beta is aimed at technical evaluators
- current validation priority is the desktop control-plane workflow, not the
  distribution experience
- the repo already provides a repeatable local run path for desktop
- promising an installer before the delivery shape is stable would overstate
  actual readiness

## Desktop Non-Promises

- installer
- signed desktop package
- `exe` / `msi` installer output
- auto-update
- packaged standalone desktop distribution
- built-in relay bootstrap

## Remote-Agent Trial Install Baseline

- remote-agent remains a repo-local source-install surface for the first public
  Beta
- the current trial install path is:
  - copy `remote-agent/` to the remote Linux host
  - `cd remote-agent`
  - `bash scripts/install-systemd-user.sh --start`
- the current long-running service shape is `systemd --user`
- the current service file and env file paths are:
  - `~/.config/systemd/user/remote-agent.service`
  - `~/.config/remote-agent/remote-agent.env`

## Automated Install Help

- create the virtualenv
- run `python -m pip install -e <workdir>`
- write base env values:
  - `REMOTE_AGENT_HOST`
  - `REMOTE_AGENT_PORT`
  - `REMOTE_AGENT_LOG_LEVEL`
  - `REMOTE_AGENT_LOG_FILE`
- render the user service file
- run `systemctl --user daemon-reload`
- run `systemctl --user enable remote-agent.service`
- start or restart the service when `--start` is passed
- attempt to ensure linger before claiming logout-safe hosting

## Manual Trial Configuration

- open `~/.config/remote-agent/remote-agent.env`
- add `REMOTE_AGENT_RELAY_ENDPOINT`
- add `REMOTE_AGENT_CONTROL_BASE_URL`
- preferably add `REMOTE_AGENT_REMOTE_NAME`
- confirm remote-to-local relay connectivity
- confirm local-relay-to-remote control connectivity
- confirm `kimi` is in PATH, or set `KIMI_BIN`

## Kimi Binary Discovery Baseline

- first: `--kimi-bin`
- second: `KIMI_BIN`
- third: PATH lookup for `kimi`
- no Linux-home provider fallback remains in shared runtime

## Minimal Remote Verification Path

1. `systemctl --user status remote-agent.service --no-pager`
2. `remote-agent sessions`
3. `remote-agent kimi start --task "..." [--kimi-bin ...]`
4. `tail -n 50 ~/.local/state/remote-agent/remote-agent.log`

This verifies the remote install surface only. The full end-to-end local
desktop + relay + approval trial path stays deferred to `P6.5-4`.

## Remote-Agent Trial Non-Promises

- full automatic env provisioning for relay / control / remote-name
- non-Linux deploy support
- PyPI package release
- OS package release
- packaged installer
- recovery implementation beyond the current contract boundary

### remote-agent

- delivered as repo source under `remote-agent/`
- installable as a Python package from the repo copy
- current deploy shape is Linux `systemd --user` with
  `scripts/install-systemd-user.sh`
- not a PyPI or system package release

### relay

- delivered as repo source under `relay/`
- started locally by the user as a FastAPI process
- required for the first public Beta
- not a cloud product and not a separately managed hosted service

## Minimal Trial Path

1. On the local machine, install relay dependencies and start `relay`.
2. On the same local machine, install desktop dependencies and start
   `desktop`.
3. Copy `remote-agent/` to a remote Linux host.
4. Install and start the remote user service.
5. Manually add `REMOTE_AGENT_RELAY_ENDPOINT`,
   `REMOTE_AGENT_CONTROL_BASE_URL`, and preferably
   `REMOTE_AGENT_REMOTE_NAME` to the remote env file.
6. Ensure `kimi --wire` is available on the remote host, or provide
   `KIMI_BIN`.
7. Run `remote-agent kimi start --task "..."` on the remote host.
8. Observe the session and any approval in the local desktop, then complete a
   local `approve / reject`.

## Minimum Prerequisites

- local Python environment able to run `relay`; use Python `3.10+` as the
  shared baseline
- local Node.js and npm able to run `desktop/`
- remote Linux with `python3`, `venv`, `systemctl --user`, and `loginctl`
- linger enabled for the remote user
- remote `kimi` CLI with `--wire`, or an explicit `KIMI_BIN`
- network reachability from remote hosts to local `relay`
- network reachability from local `relay` back to each remote-agent control
  address
- testers comfortable with manual env editing, SSH, and user-service setup

## Current Known Limitations

- `install-systemd-user.sh` does not yet provision
  `REMOTE_AGENT_RELAY_ENDPOINT`, `REMOTE_AGENT_CONTROL_BASE_URL`, or
  `REMOTE_AGENT_REMOTE_NAME`; first-Beta trial setup still needs manual env
  edits
- `desktop` is source-run only
- `desktop` does not currently expose build / package / installer scripts
- `install-systemd-user.sh` writes only the base host / port / log env and
  leaves relay / control / remote-name trial config manual
- `relay` is a manually started local process
- `watch` is single-read, not continuous follow
- `attach` is not implemented
- `stop` rejects `approval_pending` and running-turn sessions
- `relay` and `remote-agent` are still in-memory; restart does not restore the
  prior control-plane state
- no provider execution-state `resume / reattach` promise exists
