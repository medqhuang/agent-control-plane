# P1.5 Worklog

Updated: 2026-03-31
Status: Complete

This document records the closure work completed in `P1.5`.
The scope is limited to minimal state consistency and approval-response
idempotency, using only the current relay demo model defined by `README.md`
and `DEV.md`.

## Stage Goal

P1.5 closes the smallest correctness gaps in the relay core:

- keep `session` and `approval` state consistent in snapshot
- make `POST /v1/approval-response` stable under duplicate submission
- prevent duplicate events from polluting the event stream

## Completed Work

### 1. Session / Approval State Consistency

Implemented the minimal state transition rules:

- `approval.status = pending` -> `session.status = waiting_approval`
- `approval.status = approved` -> `session.status = running`
- `approval.status = rejected` -> `session.status = failed`

These rules are intentionally limited to the current single-approval demo model.
No multi-approval merge behavior was introduced.

### 2. Snapshot Consistency

`GET /v1/snapshot` continues to read directly from the in-memory stores.
Because `POST /v1/approval-response` now updates both approval and session
state, snapshot stays consistent after each decision.

Verified outcomes:

- initial snapshot:
  - approval `pending`
  - session `waiting_approval`
- after `approve`:
  - approval `approved`
  - session `running`
- after `reject`:
  - approval `rejected`
  - session `failed`

### 3. First-Write-Wins Idempotency

Implemented first-write-wins behavior for the same `request_id`.

Rules:

- first successful `approve` or `reject`:
  - HTTP `200`
  - update approval state
  - update session state
  - append exactly one event
- repeated same decision after terminal state:
  - HTTP `200`
  - return current approval state
  - do not update approval state again
  - do not update session state again
  - do not append a new event
- repeated opposite decision after terminal state:
  - HTTP `409`
  - return current state and error detail
  - do not update approval state
  - do not update session state
  - do not append a new event

### 4. Event Flow Stability

The event stream remains minimal.
For one `request_id`, the first accepted decision appends exactly one event:

```json
{
  "seq": 1,
  "type": "approval_response",
  "request_id": "approval_demo_1",
  "session_id": "session_demo_1",
  "decision": "approve",
  "approval_status": "approved"
}
```

Repeated same-decision submissions return `event = null`.
Repeated opposite-decision submissions return `409` and append no event.

## Implementation Location

The P1.5 rules land in these files:

- `relay/session_store.py`
  - approval-to-session status mapping
  - session state synchronization helper
- `relay/approval_store.py`
  - decision-to-approval status mapping
  - terminal-state detection
- `relay/main.py`
  - `POST /v1/approval-response` idempotency control
  - same-decision `200` branch
  - opposite-decision `409` branch
  - normal first-write path

`relay/event_log.py` was reused unchanged as the single in-memory event sink.

## Current Response Contract

First accepted decision:

```json
{
  "approval": {
    "request_id": "approval_demo_1",
    "session_id": "session_demo_1",
    "status": "approved",
    "kind": "command",
    "summary": "Approve the pending command for the demo session."
  },
  "event": {
    "seq": 1,
    "type": "approval_response",
    "request_id": "approval_demo_1",
    "session_id": "session_demo_1",
    "decision": "approve",
    "approval_status": "approved"
  },
  "idempotent": false
}
```

Repeated same decision:

```json
{
  "approval": {
    "request_id": "approval_demo_1",
    "session_id": "session_demo_1",
    "status": "approved",
    "kind": "command",
    "summary": "Approve the pending command for the demo session."
  },
  "event": null,
  "idempotent": true,
  "message": "decision already applied"
}
```

Repeated opposite decision:

```json
{
  "detail": {
    "message": "approval request already finalized with a different decision",
    "request_id": "approval_demo_1",
    "current_status": "approved",
    "attempted_decision": "reject"
  }
}
```
