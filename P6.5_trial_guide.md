# P6.5 Trial Operator Guide

Updated: 2026-04-01
Stage: P6.5 Public Beta Release
Current Subgoal: P6.5-5 Trial Operator Guide
Next Suggested Subgoal: P6.5-6 Issue Templates And Feedback Intake

This guide is for a small operator-led Beta trial with 2-5 people. It is not a
slide deck, a screenshot script, or an installer guide. It assumes the current
first public Beta boundary is already fixed to:

- local Windows `relay + desktop`
- remote Linux `remote-agent`
- provider scope limited to `Kimi --wire`
- source-run / source-install

It must stay truthful to the current product state:

- local `relay` is started manually
- `desktop` is source-run, not an installer
- remote `remote-agent` still needs manual env editing
- network reachability still needs explicit checks from both sides
- no `attach`, no packaged desktop installer, and no `P8` recovery promise

## 1. Trial Goal

Run one real hosted-session trial from local startup to one approval decision.

At the end of the session, the group should have verified:

- the local operator can start `relay` and `desktop`
- a remote Linux host can run `remote-agent` with `Kimi --wire`
- a hosted session reaches local `desktop` through `relay`
- one pending approval can be handled with either `Approve` or `Reject`

## 2. Participant Roles

Recommended shape for one small trial:

- 1 local operator
  - runs local `relay`
  - runs local `desktop`
  - watches the session list and pending approvals
  - clicks `Approve` or `Reject`
- 1-3 remote trial users
  - run `remote-agent` on remote Linux
  - check `kimi --wire`
  - launch one hosted session with `remote-agent kimi start --task "..."`
- optional 1 observer / note taker
  - records where the trial stalled
  - records which steps were manual
  - collects wording gaps and failure points

## 3. Preflight Checklist

### Local Windows Operator

- a local checkout of this repo
- Python available for `relay`
- Node.js and npm available for `desktop`
- local firewall / network route allows remote Linux to reach
  `http://<local-relay-host>:8000`

### Remote Linux Trial User

- a remote Linux host with a copy of `remote-agent/`
- `systemd --user` available
- `kimi` available through PATH, `KIMI_BIN`, or an explicit `--kimi-bin`
- network route allows the local operator to reach
  `http://<remote-host>:8711`

### Manual Inputs That Must Be Ready

These are still manual in the first public Beta:

- `REMOTE_AGENT_RELAY_ENDPOINT=http://<local-relay-host>:8000`
- `REMOTE_AGENT_CONTROL_BASE_URL=http://<remote-host>:8711`
- `REMOTE_AGENT_REMOTE_NAME=<unique-remote-id>`
- confirmation that remote-to-local and local-to-remote network access both work

## 4. What Is Automated Vs Manual

### Automated Today

- `remote-agent/scripts/install-systemd-user.sh --start` creates the venv,
  installs `remote-agent`, writes the base env file, renders the user service,
  enables the service, and can start it
- `desktop` reads `relay` snapshot and approval APIs once `relay` is already up
- approval decisions are forwarded from local `desktop` to `relay`, then back to
  the hosted session

### Still Manual Today

- starting local `relay`
- starting local `desktop`
- editing remote
  `~/.config/remote-agent/remote-agent.env`
- confirming both network directions are reachable
- making sure `kimi` is resolvable
- choosing a task that actually reaches an approval checkpoint

## 5. Local Operator Steps

All steps in this section are run on local Windows.

1. From the repo root, install relay requirements if needed:

```powershell
python -m pip install -r requirements-relay.txt
```

2. Start the local relay:

```powershell
python -m uvicorn relay.main:app --host 127.0.0.1 --port 8000
```

3. Optional local relay check:

```powershell
curl.exe http://127.0.0.1:8000/v1/snapshot
```

4. In a second local shell, start desktop:

```powershell
cd desktop
npm install
npm start
```

5. Keep the desktop window open. The operator does not need to launch any
   remote process from desktop. Desktop is only the local monitor and approval
   surface in this Beta.

## 6. Remote Trial Steps

All steps in this section are run on remote Linux.

1. Install and start the user service:

```bash
cd ~/agent-control-plane/remote-agent
bash scripts/install-systemd-user.sh --start
```

2. Open `~/.config/remote-agent/remote-agent.env` and manually add:

```bash
REMOTE_AGENT_RELAY_ENDPOINT=http://<local-relay-host>:8000
REMOTE_AGENT_CONTROL_BASE_URL=http://<remote-host>:8711
REMOTE_AGENT_REMOTE_NAME=<unique-remote-id>
```

3. Restart the user service after editing env:

```bash
systemctl --user restart remote-agent.service
systemctl --user status remote-agent.service --no-pager
```

4. Confirm remote Linux can reach the local relay:

```bash
curl http://<local-relay-host>:8000/v1/snapshot
```

5. Confirm Kimi discovery:

```bash
which kimi
kimi info
```

If `kimi` is not in PATH, do not describe it as auto-discovered. Use either:

- `KIMI_BIN=/path/to/kimi`
- `remote-agent kimi start --kimi-bin /path/to/kimi --task "..."`

6. Optional local-to-remote check from the operator machine:

```powershell
curl.exe http://<remote-host>:8711/healthz
```

## 7. Full Approval Demo Path

Use a remote working directory that is safe for a test write:

```bash
mkdir -p ~/acp-beta-trial
cd ~/acp-beta-trial
```

Start one hosted session with a task that is likely to ask for approval before
writing:

```bash
remote-agent kimi start --task "Create a file named beta-proof.txt in the current directory, but ask for approval before you write anything."
```

If `kimi` is not in PATH:

```bash
remote-agent kimi start --kimi-bin /path/to/kimi --task "Create a file named beta-proof.txt in the current directory, but ask for approval before you write anything."
```

Then verify on the remote host:

```bash
remote-agent sessions
tail -n 50 ~/.local/state/remote-agent/remote-agent.log
```

Then verify on local desktop:

- a new hosted session appears in the session list
- one pending approval appears in the approval list
- the operator completes one decision with `Approve` or `Reject`

Interpret the result like this:

- `Approve`: the decision is sent through `relay` back to the hosted session,
  and the remote task may continue
- `Reject`: the rejection goes through the same path and the remote task should
  not be treated as implicitly allowed

## 8. If The Task Does Not Trigger Approval

This is not automatically a desktop or relay failure.

The current provider may reach its first checkpoint without asking for approval.
When that happens:

1. confirm the session itself appeared in desktop or via `remote-agent sessions`
2. treat the transport path as at least partially working
3. rerun `remote-agent kimi start --task "..."` with a task more likely to
   request approval before file writes or other controlled actions

Recommended fallback tasks:

- `Create a file named beta-proof.txt in the current directory, but ask for approval before writing it.`
- `Create a new notes directory in the current folder, but ask for approval before any filesystem changes.`
- `Write a short text file in the current directory, but stop and ask for approval first.`

Do not document "no approval appeared" as proof that approval is broken until a
more approval-prone task has also failed.

## 9. Common Failures And Fast Triage

### No Session Appears In Desktop

- check local `relay` is still running
- re-run `curl.exe http://127.0.0.1:8000/v1/snapshot`
- check remote
  `REMOTE_AGENT_RELAY_ENDPOINT`
- check remote can really reach `http://<local-relay-host>:8000`
- inspect remote log:

```bash
tail -n 50 ~/.local/state/remote-agent/remote-agent.log
```

### Session Appears But No Approval Appears

- first switch to a more approval-prone task
- verify the provider did not simply complete its first checkpoint without an
  approval request
- use `remote-agent sessions` and the remote log to confirm the session exists

### remote-agent Service Does Not Stay Up

- run:

```bash
systemctl --user status remote-agent.service --no-pager
tail -n 50 ~/.local/state/remote-agent/remote-agent.log
```

- confirm the env file still contains valid URLs
- confirm the current user actually has `systemd --user`

### Kimi Cannot Be Found

- run `which kimi`
- run `kimi info`
- if missing, set `KIMI_BIN` or pass `--kimi-bin`
- do not treat PATH resolution as already automated packaging

### Operator Can See Session But Approval Buttons Do Not Complete The Flow

- confirm local desktop is still pointing at the correct `relay` URL
- confirm the remote control base URL is reachable from the local machine
- confirm the remote session is still alive and not already stopped

## 10. Feedback To Collect

Collect feedback immediately after the trial while the exact step is still
fresh.

For each participant, record:

- role: local operator, remote trial user, or observer
- local OS / remote OS
- Python version, Node.js version, and whether `kimi` came from PATH or
  `KIMI_BIN`
- which exact step first caused confusion
- whether the hosted session appeared in desktop
- whether an approval appeared
- whether `Approve` and `Reject` behaved as expected
- which parts were still manual but felt surprising
- the exact command or check that unblocked them
- the smallest documentation change that would have saved time

## 11. End Of Trial

At the end of one guided trial, the operator should be able to answer:

- did the local `relay + desktop` path start cleanly
- did remote Linux install and keep `remote-agent` alive with `systemd --user`
- did `Kimi --wire` start through the current discovery rules
- did the group complete at least one real `Approve` or `Reject`
- if not, was the blocker task selection, env, network, or service health

That output is the stable handoff into `P6.5-6 Issue Templates And Feedback
Intake`.
